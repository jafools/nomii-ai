## Layer 1 — Static audit

### Config — `docker-compose.yml` (SaaS) vs `docker-compose.selfhosted.yml`

| Concern | SaaS (`docker-compose.yml`) | Self-hosted | Verdict |
|---|---|---|---|
| DB password | `${DB_PASSWORD:-nomii_dev_2026}` | `${DB_PASSWORD}` (fail-fast) | **Finding #1 / HIGH** |
| JWT_SECRET | `${JWT_SECRET:-nomii-dev-secret-change-in-production}` | `${JWT_SECRET}` (fail-fast) | **Finding #1 / HIGH** |
| WIDGET_JWT_SECRET | `${WIDGET_JWT_SECRET:-}` (empty default) | `${WIDGET_JWT_SECRET}` (fail-fast) | Lower risk — signing fails loudly, but should still fail at startup, not runtime. |
| API_KEY_ENCRYPTION_SECRET | `${API_KEY_ENCRYPTION_SECRET:-}` (empty default) | `${API_KEY_ENCRYPTION_SECRET}` (fail-fast) | As above — encryption breaks silently for stored API keys. |
| DB user / DB name | `nomii` / `nomii_ai` | `knomi` / `knomi_ai` | **Finding #5 / MEDIUM** |
| Cloudflare tunnel | Top-level service (always on) | Behind `profiles: [tunnel]` | Intentional — install.sh auto-enables profile when token present. |
| Build vs image | `build: ./server`, `./client` | `image: ghcr.io/...:stable` | Intentional — SaaS builds from source, on-prem pulls prebuilt. |
| Backend port binding | `3001:3001` (exposes 0.0.0.0) | `127.0.0.1:3001:3001` (localhost-only) | Correct — SaaS fronts via Cloudflare, self-hosted expects reverse-proxy or direct access on 80. |
| `NOMII_LICENSE_MASTER` | `${NOMII_LICENSE_MASTER:-true}` (default true) | Not present (implicit false) | **Finding in passing** — anyone cloning SaaS compose for dev becomes a license-master. Should be explicit env-var, not a default. |
| `NOMII_DEPLOYMENT` | Not present | Hard-coded `selfhosted` | Good. |

**Finding #1 (HIGH) — SaaS compose has weak default secrets**

```yaml
# docker-compose.yml:12
POSTGRES_PASSWORD: ${DB_PASSWORD:-nomii_dev_2026}
# docker-compose.yml:33
JWT_SECRET: ${JWT_SECRET:-nomii-dev-secret-change-in-production}
```

`${VAR:-default}` uses the default when `VAR` is **unset or empty**. If a deploy
accidentally ships a `.env` with `JWT_SECRET=` (e.g. a template), the backend
silently runs with the dev secret. Signed tokens become forgeable by anyone
with read access to the repo.

**Fix:** change to fail-fast `${JWT_SECRET:?JWT_SECRET must be set}` matching
the self-hosted compose. `docker compose up` will refuse to start when the var
is missing or empty.

### Scripts

**Finding #2 (HIGH) — `scripts/update.sh` assumes a git clone**

```bash
# scripts/update.sh:33
if git pull origin "$(git rev-parse --abbrev-ref HEAD)"; then
```

`install.sh` creates `$HOME/nomii/` (default) with only `.env` and
`docker-compose.selfhosted.yml` — no git repo. `update.sh` would fail at
`git rev-parse` with "fatal: not a git repository".

Compounding issue: install.sh's closing banner tells users the correct upgrade
command — `docker compose pull && docker compose up -d`. So `update.sh` is
effectively dead code that will mislead anyone who finds it.

**Fix options:**
1. **Delete** `scripts/update.sh` — install.sh's printed guidance is already
   correct.
2. **Rewrite** to pull the latest `docker-compose.selfhosted.yml` from GitHub
   raw, run `docker compose pull`, then `docker compose up -d`. This is more
   complete (picks up compose-file changes between releases).

Recommend #2 — release-flow changes are exactly the kind of thing that need a
compose-file refresh.

**Finding #4 (MEDIUM) — `scripts/migrate.sh` DB defaults disagree with self-hosted compose**

```bash
# scripts/migrate.sh:11-12
DB_USER="${NOMII_DB_USER:-nomii}"
DB_NAME="${NOMII_DB_NAME:-nomii_ai}"
```

But `docker-compose.selfhosted.yml` creates `knomi`/`knomi_ai`. `migrate.sh`
silently fails on a stock self-hosted install. In practice the backend runs
migrations on startup via app code, so `migrate.sh` is only used for manual
interventions — but anyone who runs it thinking it'll work gets confusing
silent failure.

**Fix:** default to reading from `.env` (or match compose defaults).

**Finding #7 (LOW) — Migration `015b` is a non-numeric sub-migration**

`sort -V` handles it correctly, but downstream tools that expect `NNN_*.sql`
(dbmate, goose, etc.) would break. Not urgent — we rolled our own runner.

**Finding #5 (MEDIUM) — Self-hosted DB still uses pre-rename `knomi`**

| `Covenant Trust/DECISIONS_LOG.md` | Customer-private | Out of scope |

**Options:**
- **A — Full rename** (clean, risky): rename the DB user + DB name on upgrade
  via a migration script. Requires downtime or careful `pg_dump` + recreate.
  Every existing customer gets touched.
- **B — Keep `knomi` as permanent backend handle** (cheap, honest): commit to

  match.
- **C — Rename for new installs only** (messy): `install.sh` creates `nomii`
  going forward; existing customers keep `knomi`. Code has to handle both.

Recommend **B** — the rename was purely branding; the DB user name has no
customer-facing visibility. Changing it is real risk for zero user-visible
benefit. Just be consistent in internal tooling.
